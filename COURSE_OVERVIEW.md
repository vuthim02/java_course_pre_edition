#### Topics

**1.1.1 The Birth of Java (1991-1995)**
- Created by James Gosling at Sun Microsystems
- Originally called "Oak" for set-top boxes
- Renamed to Java (after Java coffee)
- Released 1995 — web applets made it explode

**1.1.2 Java's Design Goals**
```
┌────────────────────────────────────────────┐
│            JAVA DESIGN GOALS                │
├────────────────────────────────────────────┤
│ Simple        │ Familiar but cleaner than   │
│               │ C++                        │
├───────────────┼────────────────────────────┤
│ Object-       │ Everything (except          │
│ Oriented      │ primitives) is an object    │
├───────────────┼────────────────────────────┤
│ Platform      │ JVM abstraction — run       │
│ Independent   │ anywhere                    │
├───────────────┼────────────────────────────┤
│ Secure        │ No pointers, bytecode       │
│               │ verification, sandboxing    │
├───────────────┼────────────────────────────┤
│ Robust        │ Strong typing, exceptions,  │
│               │ garbage collection          │
├───────────────┼────────────────────────────┤
│ Multithreaded │ Built-in threading from     │
│               │ day one                     │
├───────────────┼────────────────────────────┤
│ Portable      │ Primitive sizes are fixed   │
│               │ regardless of platform      │
├───────────────┼────────────────────────────┤
│ High          │ JIT compilation, adaptive   │
│ Performance   │ optimization                 │
└───────────────┴────────────────────────────┘
```

#### 1.2.1 The Compilation Pipeline

```
Hello.java (Source Code)
    │ javac Hello.java
    ▼
Hello.class (Bytecode — platform independent)
    │ java Hello
    ▼
JVM loads .class file
    │
    ▼
Bytecode Verifier checks for security violations
    │
    ▼
Interpreter starts executing (slow start)
    │
    ▼
JIT Compiler identifies "hot spots" (frequently run code)
    │
    ▼
Hot code compiled to native machine code (fast!)
    │
    ▼
Native code runs directly on CPU
```

Key insight: Java is **both** compiled AND interpreted:
1. **Compiled** (javac): Source → Bytecode
2. **Interpreted** (JVM): Bytecode → execution
3. **JIT Compiled** (JVM runtime): Hot bytecode → native machine code

This hybrid approach gives Java:
- **Portability** (bytecode runs anywhere)
- **Performance** (JIT makes hot code native-fast)
- **Security** (bytecode verification)
- **Dynamic optimization** (adaptive at runtime)

#### 1.2.2 JVM Runtime Data Areas

**The Heap (Shared)**
```
                    HEAP
 ┌────────────────────────────────────────────┐
 │           YOUNG GENERATION                 │
 │  ┌────────┐  ┌────────┐  ┌──────────┐    │
 │  │ Eden   │  │  S0    │  │    S1    │    │
 │  │ (new   │  │(survi- │  │ (survi-  │    │
 │  │ objects)│  │ vor)   │  │  vor)    │    │
 │  └────────┘  └────────┘  └──────────┘    │
 ├────────────────────────────────────────────┤
 │        OLD GENERATION / TENURED            │
 │    (Objects that survived many GC cycles)  │
 ├────────────────────────────────────────────┤
 │            METASPACE (NOT in heap)          │
 │   (Class metadata, method data, bytecode)  │
 └────────────────────────────────────────────┘
```

**The Stack (Per Thread)**
```
THREAD 1 STACK              THREAD 2 STACK
┌──────────────────┐       ┌──────────────────┐
│ Stack Frame      │       │ Stack Frame      │

│ └──────────────┘ │       │ └──────────────┘ │

│ (main method)    │       │ (main method)    │
└──────────────────┘       └──────────────────┘
```

**Heap vs Stack — The Mental Model**

Think of it like a cafeteria:
- **Stack** = Your tray. Each method call adds a new compartment. Local variables are your fork and spoon. When you're done eating (method returns), the compartment is cleaned.
- **Heap** = The kitchen. Objects are cooked here. Many people can share the same dish (object references). The kitchen staff (GC) cleans up when nobody is using a dish anymore.
- **Metaspace** = The recipe book. Class definitions, method blueprints. Stays around as long as the restaurant (JVM) is open.

#### 1.2.3 Garbage Collection

**Why GC?** In languages like C/C++, you must manually free memory (`free()`, `delete`). This causes:
- Memory leaks (forget to free)
- Dangling pointers (free too early)
- Double free (free twice → crash)

Java's GC automates this: objects that are no longer reachable are automatically collected.

**Generational Hypothesis:** Most objects die young.
- 90%+ of objects are short-lived
- Focus GC effort on young objects → huge performance win

**GC Algorithms:**

| Algorithm | How It Works | When Used |
|-----------|-------------|-----------|
| **Serial** | Single thread, stop-the-world | Small apps, single CPU |
| **Parallel** | Multiple GC threads, stop-the-world | Throughput-focused apps |
| **G1** | Region-based, concurrent, predictable pauses | **Default since Java 9** |
| **ZGC** | Ultra-low latency (sub-millisecond pauses) | Large heaps, low-latency |
| **Shenandoah** | Concurrent compaction, low-pause | Large heaps, consistent response |

**How G1 GC Works (The Default):**

```
1. Heap divided into ~2048 regions (not contiguous generations)
2. Young collections: copy live objects from Eden → Survivor
3. Old collections: concurrent marking of live objects
4. Mixed collections: collect both young + most garbage-filled old regions
5. Goal: keep pause time under target (default 200ms)
```

#### 1.2.4 Bytecode — What's Inside a .class File?

```
.class file structure:
┌────────────────────────┐
│ Magic Number: 0xCAFEBABE│
├────────────────────────┤
│ Version Info (major.minor)│
├────────────────────────┤
│ Constant Pool          │ ← String literals, class names, method names
├────────────────────────┤
│ Access Flags           │ ← public, final, abstract, etc.
├────────────────────────┤
│ This Class, Super Class│
├────────────────────────┤
│ Interfaces             │
├────────────────────────┤
│ Fields                 │
├────────────────────────┤
│ Methods (with bytecode)│ ← The actual instructions!
├────────────────────────┤
│ Attributes             │ ← Line numbers, local variable table, etc.
└────────────────────────┘
```

**Example: Simple Java → Bytecode**

```java
// Hello.java
public class Hello {
    public static void main(String[] args) {
        int x = 5;
        int y = 10;
        int sum = x + y;
        System.out.println(sum);
    }
}
```

Compile: `javac Hello.java`
Decompile: `javap -c Hello`

```
Compiled from "Hello.java"
public class Hello {
  public Hello();
    Code:
       0: aload_0
       1: invokespecial #1     // Method java/lang/Object."<init>":()V
       4: return

  public static void main(java.lang.String[]);
    Code:
       0: iconst_5             // Push int 5 onto stack
       1: istore_1             // Store to local variable 1 (x)
       2: bipush 10            // Push int 10 onto stack
       4: istore_2             // Store to local variable 2 (y)
       5: iload_1              // Load local var 1 (x) onto stack
       6: iload_2              // Load local var 2 (y) onto stack
       7: iadd                 // Add top 2 stack values
       8: istore_3             // Store to local variable 3 (sum)
       9: getstatic #7         // Field System.out
      12: iload_3              // Load sum onto stack
      13: invokevirtual #13    // Method PrintStream.println
      16: return               // Return from method
}
```

**Key Insight:** The JVM is a **stack-based virtual machine**. Most operations push/pop from the operand stack. This is different from register-based machines (like physical CPUs).

#### 1.2.5 The Class Loading Mechanism

```
                         CLASS LOADER HIERARCHY
                    ┌─────────────────────────────┐
                    │    Bootstrap ClassLoader    │
                    │  loads: rt.jar, java.*      │
                    │  (native, part of JVM)      │
                    └─────────────┬───────────────┘
                                  │
                    ┌─────────────▼───────────────┐
                    │    Extension ClassLoader    │
                    │  loads: jre/lib/ext/*       │
                    │  (Java 9+: platform loader) │
                    └─────────────┬───────────────┘
                                  │
                    ┌─────────────▼───────────────┐
                    │    Application ClassLoader  │
                    │  loads: classpath           │
                    │  (your classes, libraries)  │
                    └─────────────┬───────────────┘
                                  │
              ┌───────────────────┼───────────────────┐
              │ Custom           │ Custom             │
              │ ClassLoader 1    │ ClassLoader 2      │
              │ (e.g., web app)  │ (e.g., plugin)     │
              └──────────────────┴────────────────────┘
```

**Delegation Model:** A class loader first delegates to its parent before trying to load a class. This ensures:
- Core Java classes are always loaded by the Bootstrap ClassLoader
- No one can replace core Java classes with malicious versions
- Consistent class definition across the application

#### 1.2.6 JVM Tuning Flags

Essential JVM flags every Java engineer should know:

```bash
# Memory settings
-Xms512m              # Initial heap size
-Xmx4g                # Maximum heap size (set to available RAM)
-Xss1m                # Thread stack size
-XX:MetaspaceSize=256m # Initial metaspace size
-XX:MaxMetaspaceSize=512m # Max metaspace size

# GC settings
-XX:+UseG1GC              # Use G1 GC (default since Java 9)
-XX:MaxGCPauseMillis=200   # Target max GC pause (ms)
-XX:+UseZGC               # Use ZGC (low-latency, Java 15+)
-XX:+UseShenandoahGC      # Use Shenandoah (low-latency)

# Diagnostic
-XX:+PrintGCDetails        # Print GC details
-XX:+PrintHeapAtGC         # Print heap state at GC
-Xlog:gc*                  # Unified GC logging (Java 9+)

# Performance
-XX:+AlwaysPreTouch        # Pre-touch heap pages at startup
-XX:+UseStringDeduplication # Deduplicate identical strings
-XX:+UseCompressedOops     # Compress object pointers (64-bit)
```

#### Exercises

1. **Bytecode Reading:** Write a simple Java class, compile it, and use `javap -c -verbose` to read the bytecode. Identify: magic number, version, constant pool entries, method bytecodes.

2. **Memory Visualization:** Write a program that creates objects in a loop. Use `jvisualvm` or `jconsole` to watch heap usage. Observe GC activity.

3. **Stack vs Heap:** Write a method that calls itself recursively without a base case. Observe the `StackOverflowError`. Note the stack trace.

4. **Flag Experimentation:** Run a simple app with `-Xmx50m` vs `-Xmx1g`. Use `-XX:+PrintGCDetails` to compare GC behavior.

5. **ClassLoader:** Write code to print the ClassLoader hierarchy of a simple class:
   ```java
   System.out.println(String.class.getClassLoader());
   System.out.println(MyClass.class.getClassLoader());
   ```
